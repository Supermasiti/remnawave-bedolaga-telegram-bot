# LatAm fork customizations (Monoza VPN)

This document tracks how this fork of `remnawave-bedolaga-telegram-bot`
diverges from upstream (`BEDOLAGA-DEV/remnawave-bedolaga-telegram-bot`) for
the Monoza VPN LatAm launch. It complements `CHANGELOG.md` (which is
auto-generated by release-please from upstream conventional commits and does
**not** track fork-only customizations).

The core customization is **de-russification**: the product targets Spanish
(default), English, Portuguese, Chinese and Farsi speakers, and must never
show Russian text or ruble pricing to a customer. The Telegram admin panel
and any RU/CIS-only payment provider are intentionally left in Russian,
since only the operator (Russian-speaking) ever sees them.

## Deployment notes (important ŌĆö code changes are not automatically live)

- **Bot backend**: application code is **baked into the Docker image at
  build time** (`build: .` in `docker-compose.yml`); it is *not* a bind
  mount. `docker compose restart bot` alone does **not** pick up Python
  source changes ŌĆö you must `docker compose build bot && docker compose up
  -d bot`. Only `locales/*.json` (and `data/`, `logs/`, `uploads/`) are
  bind-mounted, so locale-only edits take effect on a plain restart.
- **Cabinet frontend**: served as a static build by Caddy directly from
  `/srv/cabinet` on the host (see `/opt/remnawave/caddy/Caddyfile`, port
  `:8090`), independent of the git repo or any container. After changing
  `bedolaga-cabinet`, you must `npm run build` and copy `dist/*` into
  `/srv/cabinet` for the live site to reflect the change.

## Commits ŌĆö `remnawave-bedolaga-telegram-bot`

### `86ed34e4` ŌĆö fix: pin bot to static IPv4 172.18.0.7 on remnawave-network (2026-06-30)
Unrelated to i18n ŌĆö infra-only, and the only change in this fork to
`docker-compose.yml`. The `bot` service used to join `remnawave-network` with
whatever address Docker happened to assign, which could change on container
recreation. Pinned it to a static `ipv4_address: 172.18.0.7` so anything that
references the bot by IP (a firewall rule, a reverse-proxy upstream, an
allowlist on the RemnaWave panel side) keeps working across rebuilds/redeploys.

### `c21501fc` ŌĆö feat: full de-russification for LatAm launch + local currency display (2026-07-09)
- Removed `ru`/`ua` from selectable bot languages; `es` is now default.
- Replaced hardcoded Russian text with locale keys across every live
  client-facing handler: subscription flows, balance/payment, tickets,
  referral, gift activation, Stars payments, simple subscription.
- Added `app/services/currency_service.py`: fetches live USD exchange
  rates and shows MXN (es users) / BRL (pt users) alongside the USD
  amount for readability ŌĆö actual billing stays USD-only via Telegram
  Stars / CryptoBot.
- Deleted the bundled `ru.json` / `ua.json` locale files.
- Fixed `VERSION_CHECK_REPO` to point at the real upstream repo instead of
  a stale fork.

### `6f475983` ŌĆö fix: complete de-russification across cabinet, webapi, services, database and config (2026-07-10)
Follow-up pass covering everything the first commit didn't reach:
`app/cabinet` (the web personal-cabinet backend), `app/webapi/routes/miniapp.py`
(Telegram Mini App backend), `app/services`, `app/middlewares`,
`app/database/crud`, `app/config.py`. Highlights:
- Ban-notification templates (`BAN_MSG_*` in `app/config.py`) were 100%
  hardcoded Russian with zero locale awareness ŌĆö translated to English
  (the templates are sent directly to a real customer's Telegram DM when
  RemnaWave triggers a device-limit/WiFi/mobile-network ban).
- `subscription.status_display` (a raw Cyrillic model property) was used
  directly on the "My subscriptions" screen; replaced with a locale-aware
  helper.
- `format_period_description()` had a call site in `miniapp.py` missing
  the `language` argument, silently defaulting to Russian; fixed, and the
  function's own default changed from `'ru'` to `'en'`.
- Farsi (`fa`, an actively supported language) was grouped together with
  Russian in several formatter/status branches, so Farsi users were
  seeing Russian text ŌĆö fixed in `app/utils/formatters.py`.
- Fixed several DB **default-seed values** that leak into production rows
  on first use: tariff name, default promo group name, promo-offer
  templates, wheel-of-fortune config name, welcome text. Both the Python
  seed source *and* already-seeded rows in the live database were
  corrected (see "Production database changes" below).

### `cc0fb9c0` ŌĆö fix: default language fallback in create_user* was still 'ru' (2026-07-10)
Found by direct function testing after rebuilding the bot image (the prior
two commits had never actually been deployed ŌĆö see Deployment notes above).
`create_user` / `create_user_no_commit` / `create_user_by_email` /
`create_user_by_oauth` all defaulted `language='ru'` in their signatures,
and the shared `_normalize_language_code()` helper's own fallback also
defaulted to `'ru'`. Any caller omitting the `language` kwarg ŌĆö notably
`app/cabinet/routes/oauth.py`'s VK/Yandex/Discord/Google OAuth signup ŌĆö
silently registered the new account as Russian. Fixed at the root: the
fallback now resolves to `settings.DEFAULT_LANGUAGE`.

### `be341615` ŌĆö i18n: fill 550 zh/fa translation keys missing since the LatAm locale work (2026-07-10)
`locales/zh.json` and `locales/fa.json` had been left at pre-LatAm-launch
coverage (1959 keys) while `en`/`es`/`pt` grew to 2509 during the commits
above. Every one of those 550 missing keys (cart, balance, CryptoBot/Stars
payment flows, daily tariffs, gifts, referrals, ticket system, traffic
packages, wheel of fortune, etc.) fell back through `DEFAULT_LANGUAGE` to
Spanish rather than the user's actual language. Not a Russian leak, but a
real i18n completeness gap for Chinese and Farsi users ŌĆö backfilled to full
parity with `en.json`.

  because the component fell back to an inline Russian default string ŌĆö

  "all plans purchased," "delete subscription," and "insufficient balance
  on renewal" screens. Added proper translations to all 5 languages.
- Backfilled ~55 genuinely customer-facing `zh`/`fa` keys (dashboard stats,
  gift status labels, subscription pause/switch flows, trial banner,
  password-reset flow, support page) plus the 2 remaining `es`/`pt` gaps.
- Left the ~3500 `admin.*`/`adminDashboard.*`/`banSystem.*`/`news.admin.*`
  keys and the Slavic-only `_few`/`_many` CLDR plural forms alone (out of
  scope / not needed for non-Slavic languages).

## Production database changes (not tracked in git)

if none exists"), so fixing the Python source only prevents *future*
incorrect seeding ŌĆö rows already created before the fix needed a direct,
one-time correction. Each was backed up via `pg_dump --data-only

## Scope boundaries (intentionally left in Russian)

- Everything under `app/handlers/admin/**` and any `admin.*`/`adminDashboard.*`/
  `banSystem.*`/`news.admin.*` cabinet route ŌĆö only the operator sees these.
- Disabled RU/CIS-only payment providers and their code paths: YooKassa,
  WATA, Platega, Heleket, MulenPay, PAL24 (PayPalych), CloudPayments,
  FreeKassa, Kassa AI, RioPay, SeverPay, PayPear, RollyPay, AuraPay,
  OverPay, Etoplatezhi, Antilopay, Jupiter, Donut, Lava, Tribute. Only
  Telegram Stars and CryptoBot are enabled for this deployment.
- The disabled NaloGO Russian tax-receipt integration.
- Log messages, code comments, and internal error/debug strings ŌĆö these
  are developer-facing, not customer-facing, and were left as-is throughout.
