# Слияние с апстримом: методика и история

Разбор конфликтов при слиянии форка (дерусификация LatAm, валюты, static IP)
с апстримом BEDOLAGA-DEV/remnawave-bedolaga-telegram-bot, и точная процедура
на будущее — чтобы Hermes/DevOps не терял правки форка при `git merge upstream/main`.

## Как проверить конфликты перед реальным мержем (dry-run, безопасно, ничего не ломает)

```
git fetch upstream --tags
git checkout -b merge-dryrun-<version> main
git merge --no-commit --no-ff <upstream-tag-or-branch>
# смотрим на CONFLICT-строки в выводе
git merge --abort   # откатить, если просто хотели посмотреть
git checkout main && git branch -D merge-dryrun-<version>
```

`git diff --name-only <merge-base>..HEAD` vs `git diff --name-only <merge-base>..<upstream-tag>`,
пересечение через `comm -12` — даёт список файлов, где вообще возможен конфликт,
до реального мержа.

| `app/handlers/subscription/tariff_purchase.py` | 187 (!) | форк и апстрим **независимо** локализовали один и тот же файл (наша дерусификация LatAm vs апстримный `0951c708 feat(l10n)`) — разные имена ключей на тех же строках | все 187 хантов разрешены в пользу нашей стороны — см. методику ниже |

### `tariff_purchase.py` — как проверялось, что там не потеряна логика апстрима

Файл 6578 строк, 187 конфликтов — слишком много для ручного построчного разбора.
Вместо этого:

1. Проверено `git log <merge-base>..v3.65.1 -- <файл>` — апстрим трогал файл
   всего 5 коммитами: 1 чисто-локализационный (`0951c708`, источник всех 187
   конфликтов) + 4 с реальной бизнес-логикой:
   - `82b71311` — фикс `NameError` в `confirm_tariff_extend` (падение после
     списания денег и продления — деньги сняты, юзер ничего не видел)
   - `6ef7d0e4` — блокировка prorated instant-switch с бесплатных тарифов
   - `76aa67cb` + `0bb486e3` — конверсия живого триала при первой платной
     покупке в мульти-тарифном режиме (продовый баг, триал глушился и терял
     ссылку/устройства)
2. Все 4 логических коммита **уже вошли в файл автомерджем, без единого
   конфликта** — конфликтовали только строки, задетые локализационным
   коммитом. Проверено прямо в файле после мержа: `_converted_from_trial`,
   `TARIFF_SWITCH_RESET_FREE_DAYS` — на месте.
3. Сам локализационный коммит `0951c708` проверен построчно (diff отфильтрован
   от текстовых литералов/`.format()`/`texts.t()`) — не содержит изменений
   условий, чисел или вызовов помимо текста/вёрстки кнопок. Чистый i18n-рефактор.
4. Вывод: все 187 хантов — конкурирующие переименования ключей одной и той же
   логики, без потери функциональности. Разрешены в пользу нашей стороны
   (наши ключи уже есть и переведены во всех 5 языках: es/en/pt/zh/fa).

### Проверки после мержа
- Синтаксис всех 166 изменённых `.py`-файлов — `python3 -m py_compile`, чисто.
- Поиск кириллицы внутри `texts.t(...)` в файлах, которые правились руками —
  один preexisting-хит (`BUY_TARIFF_BUTTON`, строка 3128, был в форке ещё до
  этого мержа, не относится к мержу) — ключ есть во всех локалях, дефолт-строка
  фактически никогда не показывается пользователю. Не исправлялось — вне
  скоупа мержа.
- **Не запускался** живой regression-тест (`tests/manual/verify_derussification.py`)
  и `pytest` — локально нет venv/зависимостей (`structlog` и т.д.), а
  живой тест требует докер-контейнера бота. Это осознанно оставлено как
  отдельный шаг перед деплоем — трогать прод-контейнер молча не стал.

## Что делать перед деплоем этой ветки
```

docker exec remnawave_bot python3 /tmp/verify_derussification.py
```
(нужно сначала обновить код в контейнере на ветку `merge-upstream-3.65.1`, иначе
тест проверит то, что уже задеплоено, а не смерженную ветку).

## Рекомендации на будущее
- `ua.json` и подобные "мы отказались от X" файлы — всегда конфликт
  modify/delete, всегда решение одно и то же (`git rm`). Можно скриптовать.
- `tariff_purchase.py` — самый дорогой файл для мержей, пока форк и апстрим
  оба продолжают его переводить независимо. Разумно даже не пытаться разрешать
  автоматикой построчно — сразу применять методику выше (найти реальные
  логические коммиты через `git log -- <файл>`, проверить что они вошли
  автомерджем, взять свою сторону во всех текстовых конфликтах).
- Перед каждым будущим апстрим-релизом полезно повторить dry-run из начала
  файла — это даёт точный список файлов-кандидатов на конфликт ещё до
  реального мержа.
