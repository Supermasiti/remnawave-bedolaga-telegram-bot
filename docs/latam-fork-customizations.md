# LatAm fork customizations (Monoza VPN)

This document tracks how this fork of `remnawave-bedolaga-telegram-bot`
diverges from upstream (`BEDOLAGA-DEV/remnawave-bedolaga-telegram-bot`) for
the Monoza VPN LatAm launch. It complements `CHANGELOG.md` (which is
auto-generated by release-please from upstream conventional commits and does
**not** track fork-only customizations).

The core customization is **de-russification**: the product targets Spanish
(default), English, Portuguese, Chinese and Farsi speakers, and must never
show Russian text or ruble pricing to a customer. The Telegram admin panel
and any RU/CIS-only payment provider are intentionally left in Russian,
since only the operator (Russian-speaking) ever sees them.

## Deployment notes (important — code changes are not automatically live)

- **Bot backend**: application code is **baked into the Docker image at
  build time** (`build: .` in `docker-compose.yml`); it is *not* a bind
  mount. `docker compose restart bot` alone does **not** pick up Python
  source changes — you must `docker compose build bot && docker compose up
  -d bot`. Only `locales/*.json` (and `data/`, `logs/`, `uploads/`) are
  bind-mounted, so locale-only edits take effect on a plain restart.
- **Cabinet frontend**: served as a static build by Caddy directly from
  `/srv/cabinet` on the host (see `/opt/remnawave/caddy/Caddyfile`, port
  `:8090`), independent of the git repo or any container. After changing
  `bedolaga-cabinet`, you must `npm run build` and copy `dist/*` into
  `/srv/cabinet` for the live site to reflect the change.

## Commits — `remnawave-bedolaga-telegram-bot`

### `86ed34e4` — fix: pin bot to static IPv4 172.18.0.7 on remnawave-network (2026-06-30)
Unrelated to i18n — infra-only, and the only change in this fork to
`docker-compose.yml`. The `bot` service used to join `remnawave-network` with
whatever address Docker happened to assign, which could change on container
recreation. Pinned it to a static `ipv4_address: 172.18.0.7` so anything that
references the bot by IP (a firewall rule, a reverse-proxy upstream, an
allowlist on the RemnaWave panel side) keeps working across rebuilds/redeploys.

### `c21501fc` — feat: full de-russification for LatAm launch + local currency display (2026-07-09)
- Removed `ru`/`ua` from selectable bot languages; `es` is now default.
- Replaced hardcoded Russian text with locale keys across every live
  client-facing handler: subscription flows, balance/payment, tickets,
  referral, gift activation, Stars payments, simple subscription.
- Added `app/services/currency_service.py`: fetches live USD exchange
  rates and shows MXN (es users) / BRL (pt users) alongside the USD
  amount for readability — actual billing stays USD-only via Telegram
  Stars / CryptoBot.
- Deleted the bundled `ru.json` / `ua.json` locale files.
- Fixed `VERSION_CHECK_REPO` to point at the real upstream repo instead of
  a stale fork.

### `6f475983` — fix: complete de-russification across cabinet, webapi, services, database and config (2026-07-10)
Follow-up pass covering everything the first commit didn't reach:
`app/cabinet` (the web personal-cabinet backend), `app/webapi/routes/miniapp.py`
(Telegram Mini App backend), `app/services`, `app/middlewares`,
`app/database/crud`, `app/config.py`. Highlights:
- Ban-notification templates (`BAN_MSG_*` in `app/config.py`) were 100%
  hardcoded Russian with zero locale awareness — translated to English
  (the templates are sent directly to a real customer's Telegram DM when
  RemnaWave triggers a device-limit/WiFi/mobile-network ban).
- `subscription.status_display` (a raw Cyrillic model property) was used
  directly on the "My subscriptions" screen; replaced with a locale-aware
  helper.
- `format_period_description()` had a call site in `miniapp.py` missing
  the `language` argument, silently defaulting to Russian; fixed, and the
  function's own default changed from `'ru'` to `'en'`.
- Farsi (`fa`, an actively supported language) was grouped together with
  Russian in several formatter/status branches, so Farsi users were
  seeing Russian text — fixed in `app/utils/formatters.py`.
- Fixed several DB **default-seed values** that leak into production rows
  on first use: tariff name, default promo group name, promo-offer
  templates, wheel-of-fortune config name, welcome text. Both the Python
  seed source *and* already-seeded rows in the live database were
  corrected (see "Production database changes" below).

### `cc0fb9c0` — fix: default language fallback in create_user* was still 'ru' (2026-07-10)
Found by direct function testing after rebuilding the bot image (the prior
two commits had never actually been deployed — see Deployment notes above).
`create_user` / `create_user_no_commit` / `create_user_by_email` /
`create_user_by_oauth` all defaulted `language='ru'` in their signatures,
and the shared `_normalize_language_code()` helper's own fallback also
defaulted to `'ru'`. Any caller omitting the `language` kwarg — notably
`app/cabinet/routes/oauth.py`'s VK/Yandex/Discord/Google OAuth signup —
silently registered the new account as Russian. Fixed at the root: the
fallback now resolves to `settings.DEFAULT_LANGUAGE`.

### `be341615` — i18n: fill 550 zh/fa translation keys missing since the LatAm locale work (2026-07-10)
`locales/zh.json` and `locales/fa.json` had been left at pre-LatAm-launch
coverage (1959 keys) while `en`/`es`/`pt` grew to 2509 during the commits
above. Every one of those 550 missing keys (cart, balance, CryptoBot/Stars
payment flows, daily tariffs, gifts, referrals, ticket system, traffic
packages, wheel of fortune, etc.) fell back through `DEFAULT_LANGUAGE` to
Spanish rather than the user's actual language. Not a Russian leak, but a
real i18n completeness gap for Chinese and Farsi users — backfilled to full
parity with `en.json`.

### `a956c966` — merge: upstream v3.65.1, 150 commits behind (2026-07-19)
Fork was 150 commits behind `upstream/main`. Merged via a dedicated branch
(`merge-upstream-3.65.1`, commits `d3e65208`/`e8bf0679`) after a non-destructive
dry-run to enumerate real conflicts first. Of 33 files touched by both sides,
only 5 had actual line-level conflicts:
- `app/cabinet/routes/balance.py`, `app/cabinet/routes/subscription_modules/purchase.py` —
  trivial, took upstream's new logic (CisPay status mapping, `_persist_failed_refund`
  audit-log reason) with our English strings kept.
- `app/services/promocode_service.py` — **not trivial**: upstream added a
  combined `BALANCE_AND_DAYS` promo-code type and reordered the
  balance/days-grant blocks to prevent a double-grant on rollback. Our old
  balance-grant block (positioned before upstream's reorder) was removed
  entirely — keeping it would have double-credited balance for the new
  combined type. Upstream's surviving balance-grant block was translated to
  our `texts.t()` keys instead of its hardcoded `₽`/Russian.
- `app/localization/locales/ua.json` — modify/delete, resolved as delete
  (fork doesn't support Ukrainian). Will recur on every future merge.
- `app/handlers/subscription/tariff_purchase.py` — 187 conflicting hunks, but
  purely two independent i18n passes on the same lines (our LatAm keys vs.
  upstream's own `feat(l10n)` commit `0951c708`), not competing logic.
  Verified via `git log -- <file>` that upstream's 4 real logic commits in
  this file (extend-tariff `NameError` fix, blocked prorated instant-switch
  from free tariffs, trial→paid conversion fix) had already auto-merged
  cleanly; resolved all 187 hunks to our side.

Full conflict-by-conflict methodology: `docs/upstream-merge-notes.md`.

Regression-tested before deploy: `tests/manual/verify_derussification.py` run
in an isolated throwaway container (not the live `remnawave_bot`) against the
real DB — 174/174 checks passed, no Cyrillic detected in any of the 5
languages, including live DB rows. Deployed same day (`docker compose build
bot && docker compose up -d bot`); Alembic picked up 4 new upstream migrations
(coupons, recurrent payments, grace access, CisPay tables) cleanly.

New upstream features that landed but are **not yet adapted/tested beyond
translation**: coupons, recurrent payments, "grace access" on renewal
(disabled by default), CisPay payment provider, cabinet support WebSocket
channel. Translated, but not functionally exercised — verify before enabling
to customers.

## Production database changes (not tracked in git)

The seed functions fixed in `6f475983` are idempotent ("create default row
if none exists"), so fixing the Python source only prevents *future*
incorrect seeding — rows already created before the fix needed a direct,
one-time correction. Each was backed up via `pg_dump --data-only
--column-inserts` before the change, and every `UPDATE` was guarded with
`WHERE updated_at = created_at` so only genuinely untouched auto-seeded
rows were modified (never anything the operator had customized):

| Table | Column(s) | Before | After |
|---|---|---|---|
| `tariffs` | `name`, `description` | `Стандартный` / `Базовый тарифный план` | `Standard` / `Basic plan` |
| `promo_groups` | `name` (default group) | `Базовый юзер` | `Basic user` |
| `promo_offer_templates` | `name`, `message_text`, `button_text` (3 rows: `test_access`, `extend_discount`, `purchase_discount`) | Russian marketing copy | English equivalents |
| `wheel_configs` | `name` | `Колесо удачи` | `Wheel of Fortune` |

## Host configuration changes (not tracked in git)

### Domain migration `monoza.org` → `monozavpn.com` (2026-07-19)
Product moved to a new domain; the brand name ("Monoza VPN") was intentionally
kept unchanged — only the domain, not what customers see. RemnaWave panel and
Caddy (`/opt/remnawave/.env`, `/opt/remnawave/caddy/Caddyfile`) had already
been migrated before this was checked. Found and fixed two stale references in
`~/bedolaga-bot/.env` (host-only, `.gitignore`d, never pushed anywhere):
- `CABINET_ALLOWED_ORIGINS` — was `https://my.monoza.org`, is now
  `https://my.monozavpn.com`. CORS allow-list for the cabinet's API calls to
  the bot; a stale value here silently breaks the browser cabinet with no
  server-side error, only a browser console CORS rejection.
- `CABINET_URL` — was **empty** (`CABINET_URL=`), unrelated to the domain
  move as such — it looks like it was never set at all, on either domain.
  Used across 12+ call sites (email verification links, OAuth
  `redirect_uri`, referral/gift/guest-purchase links) — with it empty, those
  were broken in production before this fix too. Set to
  `https://my.monozavpn.com`.

Applied via `docker compose up -d bot` (env vars are read at process start,
not bind-mounted — a plain `restart` would not have picked this up).

**Open item**: the current Caddyfile has no host-matched block for either
`my.monoza.org` or `my.monozavpn.com` — `/srv/cabinet` is served on bare
`:8090` with no domain binding. Public routing for `my.monozavpn.com` is
presumably handled by the Cloudflare Tunnel (config lives in the Cloudflare
Zero Trust dashboard, not on this host — see `INFRA_HANDOVER.md`). Not
verified from this host; confirm the cabinet actually loads at
`https://my.monozavpn.com` and that the browser console shows no CORS errors.
